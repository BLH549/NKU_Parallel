而且串行在完整过程中性能基本没降（0.297 → ~0.30 μs/call），说明**串行对缓存污染不敏感，SIMD非常敏感**。

### 二、根因分析：Training缓存污染

#### 2.1 内存布局全景

完整main.cpp的执行流程：

```
Training (~27s) → Guessing + Hashing 交替 (直到10M guesses)
```

**Training阶段** 读取300万口令，构建PCFG模型，产生大量常驻内存的数据结构：

| 数据结构 | 估计大小 |
|----------|---------|
| `model::preterminals` (数千个PT对象) | 数十MB |
| `model::letters/digits/symbols` (segment的ordered_values, freqs, maps) | 数十MB |
| `model::ordered_pts` (排序后的PT副本) | 数十MB |
| 各种 `unordered_map` (preterm_freq, letters_freq等) | 数十MB |
| **模型总计** | **~100-500MB** |

**Guessing阶段** 持续访问这些模型数据：
- `Generate()` → 读取 `ordered_values` [guessing.cpp:210-213](guessing.cpp#L210-L213)
- `CalProb()` → 读取 `ordered_freqs`, `total_freq` [guessing.cpp:29-31](guessing.cpp#L29-L31)
- `NewPTs()` → 操作 `curr_indices`, `max_indices` [guessing.cpp:156-162](guessing.cpp#L156-L162)
- 优先队列排序插入 → 遍历PT对象

当累积100万guesses后进入Hash阶段时，CPU各级缓存已经被模型数据和猜测字符串填满。

#### 2.2 SIMD代码的缓存灾难

关键问题在 `MD5Hash_SIMD` 的 **优化2（批量预填充）** [md5.cpp:270-277](md5.cpp#L270-L277)：

```cpp
// 提前一次性填充所有消息
Byte** allPaddedMessages = new Byte*[count];
for (int i = 0; i < count; i++) {
    allPaddedMessages[i] = StringProcess(inputs[i], ...);
}
```

当 `count ≈ 1,000,000` 时：
1. **100万次独立堆分配** — 每次 `new Byte[]` 返回随机堆地址，100万个 ~64字节的块分散在内存各处
2. **内存占用 ~100-160MB** — 加上模型数据 ~200MB+，远超L3缓存（通常8-32MB）
3. **两次全量内存遍历** — 先填充全部100万条消息（写），再逐批计算（读）。填充阶段写入的缓存行在计算到末尾时早已被后续填充操作挤出
4. **硬件预取器失效** — `allPaddedMessages[i]` 中每个指针指向随机堆地址，CPU检测不到顺序访问模式，无法预取

#### 2.3 为什么串行不受影响

串行版本 [main.cpp:101-105](main.cpp#L101-L105)：

```cpp
for (string pw : q.guesses) {
    MD5Hash(pw, state);  // 分配 → 计算 → 立即释放
}
```

每个 `MD5Hash` 的工作集只有 **一条消息的padded buffer (~64字节) + state变量 + 栈帧**，全部可放入L1缓存（32-64KB）。即使L2/L3被模型数据污染也不受影响，因为L1足够容纳串行MD5的全部工作集。

#### 2.4 为什么SIMD受影响

SIMD版本同时需要 **4条消息的数据** 用于NEON转置加载 [md5.cpp:358-377](md5.cpp#L358-L377)：

```cpp
uint32x4_t m0 = vld1q_u32((const uint32_t*)&paddedMessages_arr[0][off]);
uint32x4_t m1 = vld1q_u32((const uint32_t*)&paddedMessages_arr[1][off]);
uint32x4_t m2 = vld1q_u32((const uint32_t*)&paddedMessages_arr[2][off]);
uint32x4_t m3 = vld1q_u32((const uint32_t*)&paddedMessages_arr[3][off]);
```

这4个buffer在预填充后分散在堆中，加载它们时发生的是 **4路随机访问**。加之模型数据占据缓存，每次加载都可能触发cache miss → 访问主存（~100ns延迟 vs L1的~4ns）。

#### 2.5 缓存乒乓效应

更严重的是 **缓存颠簸（cache thrashing）**：

1. Guessing阶段：模型热数据占据L1/L2/L3
2. Hash阶段开始：100万次 `new` + `memcpy` 将模型数据逐出缓存
3. Hash计算阶段：读取预填充消息，不断与残留的模型数据竞争缓存
4. Hash结束后回到Guessing：模型数据需要从主存重新加载

这种乒乓效应在每轮（每100万guesses）重复一次，总共约10轮。

### 四、总结

| 因素 | 对SIMD的影响 | 对Serial的影响 |
|------|-------------|---------------|
| Training数据占据L2/L3 | **高** — 4路消息加载需要更多缓存空间 | **低** — 单消息工作集仅64B，L1足够 |
| 优化2的100万次堆分配 | **极高** — 随机地址，预取器失效 | N/A |
| 优化2的双次全量内存遍历 | **高** — 后填充的数据挤出先填充的 | N/A |
| Guessing后模型的缓存残留 | **中** — 与hash数据竞争L2/L3 | **极低** — L1自治 |

**核心结论**：Training缓存污染 + 优化2的批量预填充策略 = SIMD加速比从1.78x暴跌到1.12x。去掉优化2，改为增量填充即可恢复大部分加速效果。

### 二、逻辑链第一步：串行hash为何不受缓存污染影响

从 perf 数据推算每条口令的 hash 开销：

**单独串行 O1**（来自 [md5_only_perf.md](md5_only_perf.md#L44-L45)）：
- 617M cycles / 800K calls = **771 cycles/call**
- IPC = 2.19

**完整流程串行 O1**：
- Hash time 3.02s × 2.54 GHz ≈ **7.67B cycles**（hash阶段）
- 10M 条口令 → 7.67B / 10M = **767 cycles/call**

**771 vs 767 — 几乎相等。串行hash在完整流程中的单次调用开销与单独测试无差异。**

原因：串行MD5的工作集仅 ~64 字节（单条padded message）+ state变量。每次 `MD5Hash` 内部：
```cpp
Byte *paddedMessage = new Byte[paddedLength];  // 分配
// ... 立即计算（buffer在L1中）
delete[] paddedMessage;  // 释放
```
分配后立即使用，buffer从未离开L1。**L1是私有缓存，不受L2/L3污染影响。** 即使L2/L3被训练数据填满，L1命中率依然接近100%。

### 三、逻辑链第二步：SIMD为何退化 44%

**单独SIMD O1**（[perf.md:120-135](perf.md#L120-L135)）：
- 349M cycles / 800K calls = **436 cycles/call**
- L1 miss rate: 0.007%
- L1 loads/call: 169M / 800K = 211

**完整流程SIMD O1**：
- Hash time 2.48s × 2.54 GHz ≈ **6.30B cycles**
- 10M 条口令 → 6.30B / 10M = **630 cycles/call**

**退化幅度：630 / 436 = 1.44x，即多了 194 cycles/call。**

这 194 cycles 从哪来？答案是 **优化2（批量预填充）** 造成的三级缓存雪崩。

### 四、逻辑链第三步：Page Fault 直接证据

这是 perf 数据中最关键的硬证据：

| 指标 | main_serial | main_simd | 比值 |
|------|-------------|-----------|------|
| Page faults | 64,399 | 262,181 | **4.1x** |

为什么SIMD多出197,782次page fault？

```cpp
// SIMD 优化2：一次性预填充全部100万条消息
Byte** allPaddedMessages = new Byte*[count];
for (int i = 0; i < count; i++) {
    allPaddedMessages[i] = StringProcess(inputs[i], ...);
    // ↑ 每次StringProcess调用 new Byte[] → 堆向OS申请新page
}
```

串行版本：分配→计算→立即释放。glibc malloc 复用同一块内存，堆不扩张。

SIMD版本：**100万次分配全部存活**，堆必须扩张到 ~64MB。每次 `brk()` 扩张触发 page fault。10轮hash = 10次堆扩张+收缩周期。

**计算验证：**
- 额外page faults: 262,181 - 64,399 ≈ 198,000
- 每个page fault（minor fault）成本：~500-2000 cycles（页表更新 + TLB刷新）
- 198,000 × ~1000 cycles / 10M passwords ≈ **~20 cycles/call** 来自page faults

### 五、逻辑链第四步：L3容量溢出——缓存雪崩的核心机制

Kunpeng 920 的缓存层级（典型ARM服务器）：
- L1D: 64KB per core
- L2: 512KB per core  
- L3: 24-32MB shared

**单独SIMD**：8条口令 × 64字节 = 512字节工作集 → 全部在L1内。

**完整流程SIMD**：100万条预填充buffer = **~64MB 连续分配的堆内存**。

关键时间线：

```
时刻T0: 分配buffer[0]，填入数据  → buffer[0]在L1/L2/L3
时刻T1: 分配buffer[1]，填入数据  → buffer[1]在L1/L2/L3
...
时刻T500K: 分配buffer[500K]      → 32MB L3已满
时刻T500K+1: 分配buffer[500K+1]  → 挤出buffer[0]（L3 eviction!）
...
时刻T1M: 分配完毕，64MB中前32MB已被挤出L3

时刻T1M+1: 开始计算，batch[0]访问buffer[0]
           → L1 miss → L2 miss → L3 miss → 主存访问(~100ns=250 cycles)
时刻T1M+1: batch[1]访问buffer[1]
           → 同样主存访问
...
```

**计算验证：**
- 前32MB（~500K条）被挤出L3 → 每条需从主存加载 = ~250 cycles
- 后32MB（~500K条）还在L3 → 每条L3 hit = ~30 cycles
- 平均额外延迟：(500K × 250 + 500K × 30) / 1M = **140 cycles/call** 来自缓存miss

perf数据佐证——SIMD的LLC指标更差：
- LLC-loads: SIMD 1.78-1.82B vs Serial 1.70-1.72B → SIMD多~6%
- LLC-misses: SIMD 227-259M vs Serial 169-256M → SIMD的LLC miss率12.5-14.5% vs Serial 9.8-15.0%

注意这些是**全程序**的LLC数据，包含了training（~30s）的噪音。如果只看hash阶段，SIMD的LLC miss率会远高于serial。

### 六、逻辑链第五步：为什么串行不受L3溢出影响

串行版本的访问模式：
```
for each password:
    分配1个buffer（64字节）
    立即计算（buffer在L1中，L1命中~100%）
    释放buffer
```

没有"分配-计算"之间的时间差。buffer从分配、填充到计算，整个过程在微秒级完成，数据从未离开L1。**L3是否被训练数据污染，与L1命中率无关。**

### 七、完整因果链总结

```
Training: 构建~200MB PCFG模型
    ↓
模型数据占据L2/L3缓存
    ↓
Guessing: 持续访问模型数据，L2/L3被模型热数据锁定
    ↓
```
